## 一、严重问题 (P0,需要先处理)

### 1.1 ORACLE 模式下 MPC 策略全面劣于基线
源码:`main.py:78-82` 注释明确写到 "oracle 下 MPC Cost Savings 仍为负 → BESS 经济性本身有问题",这种状态在当前 outputs 中已成事实。

`outputs/strategy_comparison.csv` 的关键数字:

| Strategy | Cost Savings | CO2 Reduction | Peak Reduction | BESS Cycles | Avg SOC |
|---|---|---|---|---|---|
| Rule-Based Peak Shaving | **+1.09%** | +0.29% | 0% | 671 | 0.303 |
| Baseline (No Storage) | 0% | 0% | 0% | 0 | 0 |
| Economic MPC | **-0.19%** | -0.17% | 0% | 21 | **0.115** |
| Robust MPC (Conformal) | -0.19% | -0.17% | 0% | 21 | 0.115 |

| DRL-SAC | -0.89% | -1.25% | **-4.01%** (峰值反增) | 1134 | 0.293 |

ORACLE 应是 MPC 的理论上界,但当前 4 种 MPC 全是负贡献,Carbon-Aware MPC 甚至比 baseline 多排碳 0.51%——这是 **逻辑反向**。Pareto 前沿因此只剩 Rule-Based 一个点,`outputs/decision_recommender.json` 5 种用户偏好全部推荐 Rule-Based,论文卖点 (MPC + Robust + Carbon + Stochastic + DRL) 全部失效。

### 1.2 MPC 在仿真第 1 小时直接把电池一次放空,之后再不循环
源码:`enhanced_ies.py:235-298` 逐步执行 MPC 第 0 步动作。

`outputs/ts_Economic_MPC.csv` 前两行:

```
2024-01-02 01:00:00, ..., bess_power=61560, soc=0.1, ...
2024-01-02 02:00:00, ..., bess_power=0,     soc=0.1, ...
```

物理核算:仿真初始 SOC=0.5,可用电量 (0.5-0.1)*162000=64,800 kWh;以放电效率 0.95 输出 = 61,560 kW·h,正好把 SOC 一次拉到 min_soc=0.1。**问题在于这个时刻是 1 AM 谷价 (0.3 元/kWh),应该是充电时段,而不是放电时段**。

`outputs/08_soc_distribution.png` 直观证实:Economic / Carbon / Robust / Stochastic 的 violin 全部死贴在 10%,几乎不可分辨;`outputs/07_daily_profile.png` 中 MPC 类 BESS Power 子图常年贴零线。MPC 把电池倒空后整年再没回填,直接证伪 "MPC 学到了价格套利" 的论文叙事。

根因猜测 (源码 `strategies.py:_ParametrizedMPC`):
- `obj_soc = 500 * |soc[H] - soc_init|` 中的 `soc_init` 每步会被更新为当前 BES 实际 SOC (`strategies.py:149`),所以一旦 SOC 落到 0.1,下一步的 MPC 终端目标也变成 0.1,**完全没有"回到合理水位"的拉力**。
- `peak_charge_weight * peak` 中的 `peak` 是 24h 窗内的最大 grid_import,而当数据集真实日峰恰好在 1-2 AM (港口夜间作业,见 `ts_*.csv` 第 2-4 行 net_demand: 49228/54615/35839),MPC 反而**用谷价电去削谷价时段的"日峰"**,既不省钱也不省排放。

### 1.3 BESS 过放电的能量被静默丢弃 (隐式 bug)
源码:`enhanced_ies.py:271-273`
```python
actual_grid = actual_demand - actual_renewable - bess_signed
if not allow_export:
    actual_grid = max(0.0, actual_grid)
```

复用上一节 `ts_Economic_MPC.csv` 的第 1 行:67,975 - 18,747 - 61,560 = **-12,332 kW**(过剩),被 `max(0, …)` 直接丢弃。

含义:BESS 实际放出 61,560 kWh,但只有 49,228 kWh 实际抵消了负荷,**剩下 12,332 kWh 既没回到电网也没回到电池**。这一损失没体现在 `cost`/`co2_kg` 字段里,也没在任何分析图中报警。建议在 `simulate_with_strategy` 中:
- 物理上限制 `bess_discharge[0] ≤ load - renewable` 之后再执行,或
- 在 IES 内部检查到过放电时回退到 `bess.discharge(load - renewable, dt)`,把多余的部分还回 SOC。

### 1.4 Carbon-Aware MPC 长期走到 `_heuristic_fallback`,碳目标完全失效
关键证据组合:

(a) `outputs/experiments/experiment_summary.json` 中 `exp04.sensitivity_scan`:
```
sensitivity in [0.5, 1.0, 2.0, 3.0, 5.0, 7.0, 10.0]
energy_cost ≡ 622217498.978
co2_tons    ≡ 544719.690
total_w_carbon ≡ 687583861.776
```
碳敏感系数从 0.5 调到 10 (20×),输出**完全相同**。

(b) `exp05` 时序 (`experiment_summary.json` 行 661-872): Carbon-MPC 求解时间 p50=0.247 ms、mean=0.275 ms,而 Econ-MPC p50=28.4 ms。CVXPY 单步 24 步 QP 不可能 0.2 ms 完成 → Carbon-MPC 每步都在走 `strategies.py:226 _heuristic_fallback`。

(c) `outputs/experiments/exp04_marginal_abatement.png`:在碳价 50-250 CNY/t 段,Carbon-Aware MPC **比 Econ MPC 多排放** 437-1637 吨 CO2;只有 300 CNY/t 才出现正减排 +10,250 t。**碳目标失效甚至反向**。

可能根因:`strategies.py:353` `cw = price/100 * sens = 1.2 * 3 = 3.6`,carbon 项 = 3.6 × Pg × intensity ≈ 2.5 元/kWh,**比电价大 2-8 倍**,使 CVXPY 目标尺度严重失衡,SCS/CLARABEL/OSQP 三层都无法求出 OPTIMAL → fallback 路径接管。

## 二、严重问题 (P1,影响论文叙事)

### 2.1 exp06 自洽性破裂: 162 MWh 不是最优容量
- `main.py:84-86` 注释:`162 MWh (P1 推荐: 从 exp06 NPV 最大化得出)`
- `experiment_summary.json` 实测:`exp06.optimal_capacity_kwh = 5000`、`optimal_NPV_CNY = -6,090,934`、`annual_savings_optimal = -318,716 元/年`
- `exp06_bess_investment_return.png` 全部 11 个候选容量 (5/10/15/20/30/50/75/100/150/200/300 MWh) **都是 NPV < 0** (红色气泡)
- `exp06_bess_optimal_sensitivity.png` 标注 "最优 5 MWh"、"NPV/Inv = -1.60",连最优档都仍亏损
- **结论:配置选了 162 MWh,但论据是错的;在当前 MPC 行为下任何容量都不经济**

### 2.2 exp07 TOU 价差越大、节省越差(逆物理直觉)
`experiment_summary.json` `exp07`:

| peak/valley | savings_pct |
|---|---|
| 2.0 | -0.554% |
| 2.5 | -0.566% |
| 3.0 | -0.575% |
| 3.5 | -0.582% |
| 4.0 | -0.587% |
| 4.5 | -0.591% |
| 5.0 | -0.595% |

价差扩大,套利空间应该**变大**,但 MPC 节省率反而**单调更差**。这印证了 1.2 的诊断:MPC 没有在做价格套利,价差扩大反而加重了它"在错误时段放电"的损失。Robust 与 Economic 行 列完全相同 (q_hat=0 退化)。

### 2.3 exp05 solver 计数全 0 + 主求解器标注矛盾
- `experiment_summary.json` `exp05.Econ-MPC.solver_counts = {OSQP:0, CLARABEL:0, default:0}`,`Carbon-MPC` 同样
- 但 `exp05_solver_summary.png` 表格里的 "Primary Solver" 列写 `OSQP (0%)`、"Steps" 列写 0
- 实际代码 `strategies.py:168-183` 求解器顺序是 SCS → CLARABEL → OSQP,SCS 没被计数;表格里 "Primary Solver" 写 OSQP 也不对;`Steps = 0` 含义不明(应该是 sim hours 计数)
- 整张表格在没读源码的情况下基本无法解读,需要修正字段含义和取值

### 2.4 Robust MPC 整体退化为 Economic MPC,论文鲁棒性卖点失效
ORACLE 模式 q_hat = 0,所以 `RobustMPCStrategy.optimize` 中 `pred_upper = pred_demand`,与 Economic MPC 完全一致:
- strategy_comparison.csv:Total Cost / Energy Cost / CO2 / Peak / Cycles / Avg SOC 全部相同 (619,574,287.86 / 542,646.582 / 21 / 0.115)
- exp03_noise_robust_advantage.png 中 "Robust 比 Economic 节省" 0.000% (4 个噪声水平差异在 ±0.001%)
- exp07 中两者每行完全一致

如果论文要讲 Robust 故事,需要**关掉 ORACLE,跑 NN 模式 + CP 校准,并保证 NN 模型有合理误差**(否则 CP 区间退化、Robust 也退化)。

### 2.5 DRL-SAC 训练未收敛、反而把峰值推高
- `strategy_comparison.csv` DRL-SAC: Peak Grid 453,108 kW 比 Baseline 435,634 kW **多 4.01%**,Cycles 1134 (比 Rule-PS 671 还多)
- `outputs/07_daily_profile.png` Avg BESS Power 子图中 DRL-SAC 在 ±10,000 kW 之间剧烈震荡,SOC 子图也在 0.1-0.6 大幅来回
- `outputs/09_timeseries.png` Grid Import 子图中 DRL-SAC 蓝线频繁突破 100k kW
- `main.py:585` `sac_steps = 30000` 看似充分,但 reward 设计未在源码注释中清晰地包含**峰值惩罚**;最终的 DRL agent 学到了"乱充乱放"而不是"削峰套利"
- **建议**:DRL 的 reward 应显式加入 demand_charge 项 + 终端 SOC 项,并且至少跑 100k 步(只有 30k 不够 SAC 收敛)

## 六、检查方法说明 (供复核)

1. **读 README.md / main.py / strategies.py / enhanced_ies.py / conformal_predictor.py / experiments.py / analysis.py** 全部源码,理清数据流;
2. **逐张读图** (40+ 张 PNG),`Read` 工具直接载入图像,与源码描述对照;
3. **核对 CSV 数值**:对 strategy_comparison / pareto_front / ts_Economic_MPC / ts_Rule-Based_Peak_Shaving / experiment_summary 等做手工算式校验(物理量纲、电量平衡、价差套利收益估算);
